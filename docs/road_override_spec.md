# Feature Spec: City Engineer Road Data Override + Interactive Map Editor

**Branch:** `feat/road-overrides` (planned)
**Status:** Approved for implementation
**Prepared:** 2026-03-27 · JOSH v3.4

---

## Problem

JOSH infers road lane counts, widths, and speed limits from OpenStreetMap tags. When OSM tags
are absent it falls back to highway-type defaults, flagging the result with
`lane_count_estimated=True` / `speed_estimated=True`. For the Berkeley hills and narrow
residential streets — exactly where fire evacuation capacity matters most — OSM data is the
least reliable. A city engineer has the authoritative field data; there is currently no way
to get that data into JOSH without editing OSM itself.

Additionally, city conditions of approval (COAs) can impose road constraints that will never
appear in OSM — e.g. "Clark Ave is egress-only per COA" — but must be modeled for accurate
ΔT calculations.

---

## Solution Overview

1. **`config/cities/{city}_road_overrides.yaml`** — versioned, auditable YAML keyed by OSM
   segment ID storing engineer-verified lane counts, widths, and speed limits.
2. **`agents/capacity_analysis.py`** — applies overrides before HCM calculations so corrected
   values flow through the entire pipeline.
3. **`agents/visualization/road_editor.py`** — generates `output/{city}/road_editor.html`, a
   standalone Leaflet map that lets an engineer click any road segment, edit its properties,
   and download the resulting YAML.
4. **`main.py`** — new `road-editor` CLI command.

## Override YAML Format

**`config/cities/berkeley_road_overrides.yaml`:**

```yaml
# City engineer street data corrections — Berkeley, CA
# Generated by JOSH Road Editor (output/berkeley/road_editor.html)
# Apply with: uv run python main.py analyze --city "Berkeley"
#
# Each entry is keyed by OSM segment ID (osmid integer, stored as string).
# Recognized fields:
#   width_meters: float    Road total paved width (meters)
#   lane_count: int        Number of travel lanes
#   speed_limit: int       Posted speed limit (mph)
#   notes: str             Reason for override / field survey notes
#   surveyed_by: str       Person or department that verified the data
#   survey_date: str       Date of field verification (YYYY-MM-DD)

overrides:
  "123456789":
    width_meters: 5.791    # 19 ft — field measured 2026-03-15
    lane_count: 1
    speed_limit: 15
    notes: "Union St access drive — below IFC §503 minimum (20 ft one-way)"
    surveyed_by: "City of Berkeley Public Works"
    survey_date: "2026-03-15"
```

The file is **optional**. If absent, capacity analysis proceeds without modification.
Stale osmid entries (road removed from OSM) log a WARNING and are skipped — non-fatal.

## Override Application: `agents/capacity_analysis.py`

### New function `_apply_road_overrides()`

```python
def _apply_road_overrides(
    roads_gdf: gpd.GeoDataFrame,
    city_slug: str,
    base_dir: Path,
) -> gpd.GeoDataFrame:
    """
    Apply city engineer corrections from config/cities/{city}_road_overrides.yaml.

    Called at the top of analyze_capacity() before _apply_hcm_capacity().
    Overrides lane_count, speed_limit, and/or width_meters for specific osmids.
    Clears the corresponding _estimated flag for overridden fields so audit trail
    shows "engineer-verified" rather than "estimated from OSM default".
    No-ops (returns gdf unchanged) if the override file does not exist.
    """
```

**Logic:**

1. Resolve `config/cities/{city_slug}_road_overrides.yaml` — return unchanged if absent
2. Build `{str(osmid): row_index}` lookup from `roads_gdf["osmid"]`
3. For each entry in `overrides:`:
   - `lane_count` → patch value, set `lane_count_estimated = False`
   - `speed_limit` → patch value, set `speed_estimated = False`, set `speed_inferred_from_width = False`
   - `width_meters` → patch value (no separate estimated flag for width)
4. Log: `"Road overrides: N/M segments updated from berkeley_road_overrides.yaml"`
5. Unmatched osmids → `logging.warning(...)` — stale override, non-fatal

### Call site in `analyze_capacity()`

Insert before `_apply_hcm_capacity()`:

```python
# Apply city engineer road data corrections (if any)
if data_dir is not None:
    base_dir = data_dir.parent.parent   # data/{city}/ → project root
    city_slug = city_config.get("city_slug", city_config.get("city", "unknown"))
    roads_gdf = _apply_road_overrides(roads_gdf, city_slug, base_dir)
```

`city_slug` is derived from `city_config`. `data_dir` is already an optional parameter of
`analyze_capacity()`.

### Road segment color coding

| Color | Meaning |
|-------|---------|
| `#adb5bd` gray (0.3 opacity) | Both lane_count and speed_limit are OSM-measured |
| `#fd7e14` orange | Speed estimated (width-inferred or highway default) |
| `#ffc107` yellow | Lane count estimated (highway default) |
| `#dc3545` red | Both estimated — no OSM lane or speed data |
| `#28a745` green (2 px) | Has an active override loaded in the editor |

"Needs Review" filter hides all gray segments — only estimated and overridden remain.

### Input notes

- **Width input**: Accepts feet (auto-converts to meters: `ft × 0.3048`). Shows current
  ft equivalent below the input if `width_meters` is present (`{w_m / 0.3048:.1f} ft`).
- **Width → speed inference preview**: Below the width field, shows the IFC §503 tier that
  would apply: *"← 19 ft: IFC §503 substandard → 15 mph inferred"*. Updates live as user types.
- All inputs optional — only non-empty fields are written to the YAML.

### JavaScript state management

```javascript
const edits = {};   // { "osmid_string": { width_meters, lane_count, speed_limit,
                    //                     notes, surveyed_by, survey_date } }
```

Stored in `localStorage[`josh_road_overrides_${CITY_SLUG}`]` so edits survive page reload.

### "Download YAML" (client-side, no server)

```javascript
function downloadYAML() {
    let yaml = YAML_HEADER;   // schema comment block embedded from Python
    yaml += "overrides:\n";
    for (const [osmid, fields] of Object.entries(edits)) {
        yaml += `  "${osmid}":\n`;
        for (const [k, v] of Object.entries(fields)) {
            yaml += `    ${k}: ${JSON.stringify(v)}\n`;
        }
    }
    const blob = new Blob([yaml], {type: "text/yaml"});
    const a = document.createElement("a");
    a.href = URL.createObjectURL(blob);
    a.download = `${CITY_SLUG}_road_overrides.yaml`;
    a.click();
}
```

### "Load existing YAML"

`<input type="file" accept=".yaml,.yml">` triggers js-yaml parse, pre-fills `edits` from
`data.overrides`, and re-colors affected segments green.

### GeoJSON embedding

- Only `is_evacuation_route == True` segments (Berkeley: 1,659 × ~350 B ≈ 580 KB — acceptable)
- Each feature carries: `osmid`, `name`, `highway`, `road_type`, `lane_count`,
  `lane_count_estimated`, `speed_limit`, `speed_estimated`, `speed_inferred_from_width`,
  `width_meters`, `effective_capacity_vph`
- Bucketed by color tier → one `L.geoJSON()` call per tier (same pattern as `demo.py`)

### CDN dependencies

| Library | Version | Size | Purpose |
|---------|---------|------|---------|
| Leaflet | 1.9.3 | ~40 KB | Map rendering (already used in demo_map.html) |
| js-yaml | 4.1.0 | ~50 KB | Client-side YAML parse for "Load existing YAML" |

## `main.py` — New `road-editor` Command

```python
@cli.command("road-editor")
@click.option("--city", required=True, help='City name (must match a prior analyze run)')
@click.option("--state", default="CA", show_default=True)
def road_editor(city: str, state: str):
    """
    Generate an interactive road data editor map.

    Click any road segment to correct its lane count, width, or speed limit.
    Download the resulting YAML to config/cities/{city}_road_overrides.yaml
    and re-run `analyze` to apply the corrections.

    Requires a prior `analyze` run (reads data/{city}/roads.gpkg).

    Output: output/{city}/road_editor.html
    """
```

**Command body:**

1. Load config, verify `data/{city}/roads.gpkg` exists (fail clearly if not)
2. Load `roads_gdf` from `roads.gpkg`
3. Merge `effective_capacity_vph` from `output/{city}/routes.csv` on `osmid` (left join —
   enriches display only, not required)
4. Resolve overrides path: `config/cities/{city_slug}_road_overrides.yaml`
5. Call `create_road_editor(city, roads_gdf, config, city_config, overrides_path, output_path)`
6. Print: `Road Editor: output/{city}/road_editor.html`
7. Print: `After editing: copy downloaded YAML → config/cities/{city_slug}_road_overrides.yaml`
8. Print: `Then re-run: uv run python main.py analyze --city "{city}"`

## Notes

- **Width vs. lanes vs. speed priority:** Width drives speed inference only when `maxspeed`
  is absent (see `_resolve_speed` three-tier logic). An explicit `speed_limit` override always
  wins over width inference.
- **COA modeling:** Conditions of approval that prohibit a road from being used for ingress
  (e.g. Clark Ave egress-only) require a `lane_count: 0` override rather than a speed
  correction. The ΔT engine will exclude zero-capacity segments from exit routing.
  Document this in the `notes:` field.
- **roads.gpkg purity:** The override is applied in `capacity_analysis.py`, not written back
  to `roads.gpkg`. The GeoPackage always reflects raw OSM data. This means overrides are
  re-applied on every `analyze` run without needing `--refresh`.
- **Audit trail:** The `notes`, `surveyed_by`, and `survey_date` fields are stored in the
  YAML for legal chain-of-custody documentation. They do not affect calculations.
